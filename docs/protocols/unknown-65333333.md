# Unknown BLE Family — Vanity UUID `65333333-A115-11E2-9E9A-0800200CA100`

## Overview

A previously-uncatalogued BLE device family advertising on a hand-crafted
vanity service UUID. The UUID is **UUIDv1 (time-based)** and its embedded
timestamp resolves to **2013-04-09 13:00:03 UTC** — predating iBeacon by
about three months. No public reference (GitHub / Sourcegraph / Theengs /
OpenMQTTGateway / `uuid.pirate-server.com` community DB) matches the UUID,
so the family is catalogued here as `vendor: Unknown` and a fingerprint-only
parser is provided.

## Fingerprint

### Service UUID

| UUID | Notes |
|------|-------|
| `65333333-A115-11E2-9E9A-0800200CA100` | UUIDv1, minted 2013-04-09 |

UUID analysis:

- **Time-low / time-mid / time-hi-and-version** decode to early 2013.
- **Node** bytes `08:00:20:0C:A1:00` are the legacy **Sun Microsystems
  OUI** (registered 1998). The multicast bit is *clear*, so this is
  presented as a "real" hardware address — but nobody generated UUIDs on
  a Sun workstation in 2013. The node is a well-known
  **example/default value** copy-pasted from a `uuidgen` reference
  implementation (the same value appears in countless 2010–2013 tutorial
  snippets).
- **Leading 4 bytes** `0x65 0x33 0x33 0x33` = ASCII `"e333"` — vanity,
  not the random output of `uuidgen`.

The combination (vanity prefix + Sun-default node + 2013 timestamp) is
diagnostic of a hand-crafted UUID — typical of an early-2013 reference
design, hobbyist firmware, or a one-off OEM build whose author copied a
UUID template instead of generating one.

### Manufacturer Data (when present)

Two 6-byte payloads observed across two units:

```
31 01 00 00 00 00      (default / uninitialised)
31 01 CD 56 57 AB      (provisioned, 4-byte unit-specific tail)
```

- **Bytes 0–1**: constant `31 01`. Wire-encoded as the BLE manufacturer
  data CID this would be `0x0131` LE = **Cypress Semiconductor** per SIG.
  That's plausible (the UUID looks like a Cypress reference-design
  artefact), but it could equally be ASCII `"1"` followed by a sub-type
  byte and not a CID at all. The parser surfaces both interpretations.
- **Bytes 2–5**: per-unit identity / state. `00 00 00 00` is the
  uninitialised value; `CD 56 57 AB` is a provisioned unit's identifier.

## Identification

- **Primary**: service UUID `65333333-A115-11E2-9E9A-0800200CA100`.
- **Secondary** (informational): mfr-data leading `0x31 0x01` confirms the
  family is in its expected state shape.
- **Device class**: `unknown` — the parser is fingerprint-only.

## What We Can Parse from Advertisements

| Field | Source | Notes |
|-------|--------|-------|
| Family flag | service UUID | "We've seen this family before" |
| Frame type (`0x31`) | mfr byte 0 | constant across observed captures |
| Subtype (`0x01`) | mfr byte 1 | constant in observed payloads |
| Payload (bytes 2–5) | mfr | 4 bytes of per-unit data; semantics unknown |
| `uuid_generated_at` | UUID v1 timestamp | `2013-04-09T13:00:03Z` |

## What We Cannot Parse

- Vendor / brand / product class — no labelled specimen yet.
- Semantics of the 4-byte payload (counter? truncated MAC? sensor word?).
- Whether the device is a sensor, beacon, or development board.

## Stable Identity

There is no per-unit identifier embedded in the *advertisement* beyond the
4-byte payload, and that payload varies between observations of the same
physical emitter (one of the captures is the uninitialised zero state).
We therefore anchor identity on the BLE MAC for now —
`stable_key = unknown_65333333:<mac>`. Re-grouping across MAC rotations
will need richer payload sampling in the future.
