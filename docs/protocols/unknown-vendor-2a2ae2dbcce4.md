# Unknown Vendor — UUID-v1 Cluster, Node `2A2AE2DBCCE4`

## Overview

A persistent BLE emitter cluster whose 128-bit service UUIDs share a
distinctive structure: **all of them are RFC 4122 version-1 (time-based)
UUIDs generated on the same machine**, identifiable because they share
the **node** field `2A2AE2DBCCE4` and a matching 2019-era timestamp epoch.

Two surface forms have been captured in `research/adwatch_export 17.json`
(2026-06-03 capture, 1,530 combined sightings across 4 distinct devices):

| Surface form | UUID | Carries | Sightings (export 17) |
|---|---|---|---|
| Service-data key paired with `0xFFF5` adv UUID | `5810BBC0-B499-11E9-A2A3-2A2AE2DBCCE4` | constant 3-byte payload `01 02 00` | 1,499 (2 devices) |
| Standalone 128-bit advertised UUID | `089810CC-EF89-11E9-81B4-2A2AE2DBCCE4` | (sometimes a 12-hex local name, e.g. `84E585CE4724`) | 31 (2 devices) |

This parser catalogues the cluster as `vendor: Unknown` so the devices can
be counted, grouped, and flagged for ground-truth follow-up; it does **not**
attempt to attribute a product family. The interesting forensic signal is
the shared UUID-v1 generator node — strong evidence of a single vendor's
toolchain, not two unrelated devices that happen to advertise similar UUIDs.

## How UUID-v1 Reveals a Shared Origin

RFC 4122 §4.2 specifies UUID v1 layout:

```
time_low - time_mid - time_hi_and_version - clock_seq - node
```

Both observed UUIDs have:

- `time_hi_and_version = 11E9` → version nibble `1`, time-high `1E9` (mid-2019 epoch)
- `node = 2A2AE2DBCCE4` — the 48-bit node field

The node is supposed to be the generating host's IEEE EUI-48 MAC, but
`2A:2A:E2:DB:CC:E4` has the **locally-administered bit** set (LSB of the
high byte: `0x2A = 0010 1010` → bit 1 = 1), so it's a randomly-synthesized
node, not a real OUI. That's what `uuidgen` produces on systems that don't
expose a real MAC (or on a CI box that generates a fresh node per build).
The fact that **both** UUIDs share this random 48-bit value means they
were generated by the **same** `uuidgen` / `uuid` library invocation on
the **same** machine, in the same epoch. That is a strong "same vendor /
same firmware codebase" signal.

What we still can't conclude without ground truth:

- Whether the two forms are the same product (one device, two service
  endpoints) or different SKUs from the same vendor.
- What the 3-byte service-data payload `01 02 00` means (version /
  type / state?). It's static across all 1,499 sightings, so it's most
  likely a fixed announcement, not telemetry.
- Why two SIG-reserved short UUIDs (`0xFFF5`) accompany the 128-bit
  form — `0xFFF0–0xFFFF` is the BT SIG experimental range, so vendors
  often park there during prototyping.

### What We Can Parse from Advertisements

| Field | Source | Notes |
|-------|--------|-------|
| Vendor | hard-coded | `Unknown` |
| Vendor cluster | hard-coded | `uuidv1_node_2a2ae2dbcce4` |
| Adv shape | derived from match | `service_data_5810bbc0_with_fff5` or `service_uuid_only_089810cc` |
| Service-data payload | `serviceData[5810BBC0…]` (surface A) | preserved verbatim for forensic analysis |
| Local name | `localName` if present | surface B only |
| Verification hint | hard-coded | active-scan in nRF Connect; resolve public MAC OUI; GATT-connect for Device Information Service (0x180A) |

### What We Cannot Parse from the Advertisement

- Vendor / product identification.
- The semantics of the 3-byte `01 02 00` service-data payload — static
  across captures rules out telemetry; could be version / type / state
  but is opaque without firmware reversing.

## Stable Identity

Per-MAC anchoring; the cluster has no embedded per-unit signal that
survives MAC rotation:

```
stable_key = vendor_2a2ae2dbcce4:<mac>
```

## Candidate Hypotheses

None confirmed; documented to aid future triangulation.

1. **A pair of internal-development / prototype boards** — the SIG
   experimental UUID range, the static service-data payload, and the
   randomly-generated UUID-v1 node are all consistent with firmware
   that was never re-skinned for production.
2. **A vendor that ships multiple SKUs from a single codebase** where
   each SKU registers a different 128-bit service UUID but the
   generator's node is shared.

## How to verify (under 2 minutes)

1. **MAC OUI lookup** — find the device in nRF Connect on Android (iOS
   hides MACs) and resolve the OUI. A recognised IoT OEM OUI would
   collapse this from "unknown vendor" to "unknown product from <OEM>".
2. **Active scan** — many firmwares put richer info in the scan
   response. The 12-hex local name on surface B might expand.
3. **GATT connect** — read Device Information Service (0x180A) for
   `Manufacturer Name` and `Model Number`. A vendor string is
   definitive.
